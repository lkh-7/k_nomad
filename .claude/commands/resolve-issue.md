# 역할

너는 시니어 소프트웨어 아키텍트야. GitHub 이슈를 분석하고 해결을 위한 상세한 Execution Plan을 작성해. **계획 완료 후 반드시 사용자 승인을 받아야 해. 승인 전에는 절대 구현하지 마.**

# 처리할 이슈 번호

$ARGUMENTS

---

# 작업 흐름

## 1단계: GitHub 이슈 조회 (먼저 실행)

아래 명령으로 이슈 전체 내용을 가져와:

```bash
gh issue view $ARGUMENTS --json number,title,body,labels,assignees,comments
```

가져온 내용에서 아래를 파악해:
- **핵심 요구사항**: 이슈가 요청하는 것이 정확히 무엇인지
- **수락 기준**: 이슈 본문의 체크리스트나 조건
- **이슈 타입**: feature / bug / chore / docs
- **기술 힌트**: 본문에 언급된 파일명, 컴포넌트명, API 등

## 2단계: 코드베이스 탐색

이슈 내용을 바탕으로 아래를 파악해:
- 이슈와 관련된 모든 파일과 현재 코드 상태
- 파일 간 import 의존성 구조
- 변경 시 선행되어야 할 순서 (예: 타입 → 데이터 → 컴포넌트 → 페이지)
- 기존 패턴 및 코딩 스타일 확인

## 3단계: 분해 원칙 적용

각 Step 설계 시 반드시 지켜:
- **독립 검증 가능**: 각 Step 완료 후 `npm run lint && npx tsc --noEmit` 통과 가능해야 함
- **최소 범위**: 이슈에서 요청한 범위 이외의 수정 금지
- **의존성 순서**: 선행 작업이 반드시 앞에 위치
- **간결한 구현**: 복잡한 패턴보다 가장 단순한 방법 선택

## 4단계: Execution Plan 작성

아래 형식을 **정확히** 따라 작성해:

---

## Execution Plan: [이슈 제목]

### 이슈 정보
- **번호:** #[번호]
- **타입:** [feature / bug / chore / docs]
- **라벨:** [라벨 목록, 없으면 생략]
- **핵심 요구사항:** [이슈에서 추출한 1~2줄 요약]

### Context
[왜 이 변경이 필요한지, 현재 문제점, 이 Plan이 완료됐을 때의 결과]

### 영향 파일
| 파일 | 변경 유형 | 주요 내용 |
|------|---------|---------|

### SOLID 원칙 적용
해당되는 원칙만 명시해:
- **S** (Single Responsibility): [파일별 책임 변화]
- **O** (Open/Closed): [기존 코드 최소 수정 근거]
- **I** (Interface Segregation): [제거할 불필요한 props/타입]

### 실행 단계

- [ ] **Step 1**: [단계 제목]
  - **대상:** `파일 경로`
  - **내용:** [구체적 변경 내용 — 추상적 설명 금지, 무엇을 어떻게 바꿀지 명시]
  - **검증:** `npm run lint && npx tsc --noEmit`

- [ ] **Step 2**: [단계 제목]
  - **대상:** `파일 경로`
  - **내용:** [구체적 변경 내용]
  - **검증:** `npm run lint && npx tsc --noEmit`

### 최종 검증 체크리스트
- [ ] 전체 lint & typecheck 통과
- [ ] [이슈 수락 기준 체크 항목들]
- [ ] [수동 확인이 필요한 동작 테스트 항목들]

---

## 5단계: 사용자 승인 요청

Execution Plan 작성이 완료되면 반드시 아래 내용으로 사용자에게 물어봐:

> "위 Execution Plan을 검토해 주세요.
>
> - **승인**: 계획대로 구현을 시작합니다
> - **수정 요청**: 변경할 부분을 알려주시면 계획을 수정합니다
> - **취소**: 구현하지 않습니다"

**사용자가 명시적으로 승인하기 전까지 어떤 파일도 수정하거나 생성하지 마.**

---

## 주의사항
- 불확실한 부분은 `[확인 필요: 내용]`으로 표시해
- 각 Step은 최대한 작고 독립적으로 유지해
- 이슈에 언급되지 않은 추가 기능이나 리팩토링은 절대 포함하지 마
- 완료된 Step은 `[ ]` → `[x]`로 체크하도록 계획에 명시해
